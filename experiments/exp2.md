# Experiment 2: Multi-Hop Reasoning Quality

## Objective

This experiment compares **Synthology** against the **UDM (Unguided Deductive Materialization)
baseline** on the Family Tree ontology under a matched-budget protocol. Both generators are
configured to produce comparable input-graph sizes (~200 K input facts); a post-hoc balancing
script trims samples until the base-fact and inferred-target caps are met.

The central question is whether Synthology's targeted backward-chaining produces meaningfully
deeper inference chains than unguided forward-chaining materialization, and whether that
structural advantage translates into better RRN learning quality.

**Hypotheses:**

- Synthology will produce a substantially heavier tail of deep (≥ 3-hop) inferences, while
  UDM will be dominated by shallow (1–2-hop) facts generated by combinatorial rule firing on
  an unstructured ABox.
- The UDM baseline will fail to materialize any targets for the most structurally demanding
  predicates (e.g. `secondUncleOnceRemovedOf`) because unguided ABox generation rarely
  satisfies their simultaneous preconditions.
- The RRN trained on Synthology data will outperform the UDM-trained RRN on all primary
  metrics (PR-AUC, FPR), because the model must learn genuine multi-hop reasoning rather
  than shallow heuristics.

**Note on the UDM ABox:** Experiment 2 uses fully **random** ABox generation for the UDM
baseline — the deliberate worst case for the pipeline (no ontology-aware heuristics). This
choice isolates the structural advantage of backward-chaining under the hardest conditions
for UDM. Experiment 3 re-runs the comparison against a strong benchmark-provided ABox to
test the gap from the opposite direction.

**Primary metrics:** PR-AUC (↑) and FPR (↓), evaluated on a frozen Synthology-generated
test set shared by both models.

## Scientific Guardrail

> **Do not skip this section before running.**

Exp 2 is a **matched-budget comparison**:

- Shared base-fact cap and shared inferred-target cap across both generators.
- Matched split sample counts and matched split positive/negative target counts.
- Deep-hop distributions are **not** forced to parity; they are reported as diagnostics
  alongside model results and are expected to differ structurally.

## Experiment Mechanics

1. Generate a frozen deep test set once and reuse it for both models.
2. Generate the UDM baseline and Synthology training datasets independently.
3. Apply the matched-budget balancing script to align base-fact and inferred-target counts
   to within ~10 % across both generators.
4. If strict structural parity cannot be achieved within practical compute budgets, report
   the residual mismatch explicitly.
5. Train one RRN on the UDM baseline dataset and one on the Synthology dataset, using an
   identical training configuration for both.
6. Compare model metrics (PR-AUC, AUC-ROC, FPR, F1) and data diagnostics (hop distribution,
   predicate coverage, topology statistics), with explicit deep-hop reporting.

**Why the UDM baseline uses single-pass Jena in Exp 2:**
Standard Jena closure is fixed-point materialization and does not preserve derivation-depth
metadata natively. The current baseline pipeline materializes once per sample (single-pass)
and computes hop depth via a custom derivation-tracing pass over the materialization output.
This introduces additional runtime overhead beyond the closure computation itself.

## Execution Commands (in order)

### 1. Freeze the Exp 2 deep test set

```bash
uv run invoke exp2-generate-gold-test
```

### 2. Generate matched-budget train/val data for both arms

```bash
uv run invoke exp2-balance-datasets --fact-cap=<Nf> --target-cap=<Nt>
```

**Parameter notes:**

- `fact_cap` is required; `target_cap` is optional (each generator uses its own config
  default when omitted).
- `baseline_base_facts` overrides `generator.base_relations_per_sample` for the baseline.
- `synthology_proof_roots` overrides `generator.proof_roots_per_rule` for Synthology.
- Current config defaults:
  - Baseline: caps are `null` in `configs/udm_baseline/exp2_baseline.yaml` (no trimming
    unless explicitly set).
  - Synthology: `train_fact_cap=180000`, `train_target_cap=140000`,
    `proof_roots_per_rule=8` in `configs/ont_generator/exp2_synthology.yaml`.
- Generator defaults differ; explicitly setting both caps is recommended for fair comparisons.

**HPC preset** (recommended for reproducible matched-budget runs):

```bash
bsub < jobscripts/exp2-balance-datasets.sh
```

This submits:

```bash
uv run invoke exp2-balance-datasets-hpc --config-path=configs/experiments/exp2_balance_hpc.yaml
```
