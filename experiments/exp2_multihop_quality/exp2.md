# Experiment 2: Multi-Hop Reasoning Quality

## Objective

Experiment 2 tests the paper's central structural claim for the Family Tree ontology: even after enforcing deep-path parity on the UDM baseline, data generated by Synthology yields better downstream multi-hop reasoning performance. In current HPC runs, strict parity has proven difficult to achieve repeatedly within fixed walltime, so matched-budget balancing is documented as a practical fallback fairness protocol.

Paper traceability:

- Methodology framing: paper section "Experiment 2: Multi-Hop Reasoning Quality".
- Baseline fairness design: paper section "The Unguided Deductive Materialization (UDM) Baseline", especially the parity-enforcement protocol.
- Expected interpretation: results/discussion text around parity-enforced comparisons and deep-path structural quality.

## Experiment Mechanics (What Is Being Validated)

1. Generate a frozen deep test set once and reuse it for both models.
2. Generate baseline and Synthology training datasets under comparable budget constraints.
3. Enforce baseline parity with retry-loop generation until deep and structural criteria are satisfied.
4. If parity does not converge within practical compute budgets, run matched-budget balancing (shared train fact/target caps) and report that strict parity remained unattained.
5. Train one RRN on baseline and one RRN on Synthology with equivalent training protocol.
6. Compare model metrics and data diagnostics, with explicit deep-hop reporting.

Why baseline Jena is single-pass in Exp2:

- Standard Jena closure is fixed-point materialization and does not preserve derivation depth metadata natively.
- The current baseline pipeline materializes once per sample (single-pass closure) and reads hop-depth information from the Jena materialization output path used in the generator code.


Default notes for `exp2-balance-datasets`:

- `fact_cap` is required.
- `target_cap` is optional; if omitted, each generator uses its own config default.
- `baseline_base_facts` overrides `generator.base_relations_per_sample` for baseline.
- `synthology_proof_roots` overrides `generator.proof_roots_per_rule` for Synthology.
- Current generator defaults differ, so explicitly setting both caps is recommended for fair comparisons.

## High-Value Overrides (When Reviewer Asks For Sensitivity)

Baseline generation caps:

```bash
uv run invoke exp2-generate-baseline --fact-cap=250000 --target-cap=200000 --base-facts-per-sample=18
```

Synthology generation caps:

```bash
uv run invoke exp2-generate-synthology --fact-cap=250000 --target-cap=200000 --proof-roots-per-rule=10
```

Parity strictness example:

```bash
uv run invoke exp2-parity-loop \
  --max-attempts=400 \
  --min-deep-hops=3 \
  --tolerance-pct=10.0 \
  --deep-count-mode=tolerance
```

Default note:

- The task wrapper default tolerance is 10% to favor convergence speed; tighten this value for stricter parity sensitivity checks.
- Even with tolerance mode, parity may remain hard to satisfy under a 24h walltime. Treat matched-budget balancing as the operational fallback and clearly label the comparison as budget-matched rather than parity-matched.

## Expected Results (What Must Be Reported)

Data-level outputs to include:

- Deep-path count target and achieved parity traces (`d >= 3`).
- Time-to-parity metrics (baseline cumulative runtime vs Synthology generation runtime).
- Depth-bucket diagnostics (at minimum: total, hop=0, hop=1, hop=2-3, hop>=3).
- Structural parity indicators (nodes, edge density, inferred share, target ratios as configured).

Model-level outputs to include:

- `PR-AUC`, `AUC-ROC`, `F1`, and `FPR` for baseline-trained and Synthology-trained RRN.
- Same test split, same evaluation protocol, same frozen Exp2 deep test reference.

Primary logging locations:

- Run archives: `reports/experiment_runs/...`
- Dataset outputs: `data/exp2/...`
- Paper plots: `reports/paper/...`
- Model training runs: W&B + RRN run archives.
